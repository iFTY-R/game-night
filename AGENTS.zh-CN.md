# Claude 仓库说明

## Git 提交

- 每次提交前，除了使用 `commit-message-convention` 技能，还必须检查 `git status --short`、`git diff --staged --name-only`，并在必要时查看 patch 级 diff，确认本次提交只包含当前任务相关改动。
- 默认不要提交无关文件、临时文件、误产生的纯格式化结果、构建产物或未跟踪目录；如果当前改动包含多个意图，拆分为多个 commit。
- 持续使用 Conventional Commits。除非用户明确要求英文，否则 commit message 默认使用中文。subject 要清楚表达本次提交的主要目的，避免使用 `update`、`fix issue`、`misc` 这类模糊描述。
- commit message 默认追求简洁。对于较小或单一目的的改动，优先只写 `subject`；只有在“不写清楚原因、约束、风险或影响范围就容易误解”时，再补充 `body`。
- 需要写 commit body 时，也要保持短小、可快速扫读，只写具体 bullet。优先控制在 1-3 条，避免重复 diff 一眼可见的信息，也不要写成长段铺垫、过程化描述或注水解释。
- 提交前执行与改动范围匹配的验证命令。至少要对当前改动涉及的类型检查、测试或构建给出可交代的验证结果。
- 如果仓库提供了 commit message lint 脚本，提交前先执行校验。

## 代码注释

- 编写、修改或检查代码注释时，必须主动扫描常量、重要变量、方法/函数、关键分支、异步流程、生命周期清理、跨组件或跨上下文联动。默认应为这些目标补充注释，帮助维护者无需沿调用链反推即可理解代码意图。
- 注释必须解释意图、约束、副作用、单位、边界含义、数据归属、更新流程、业务影响、兼容要求，或为什么必须采用当前实现方式。不要只是复述名称、类型或下一行代码已经明显表达的操作。
- 常量应说明用途、单位、来源、调参原因、兼容含义、边界影响，或为什么不能随意调整。只有明显自解释的展示文案、枚举标签和一次性局部别名可以跳过。
- 重要变量应说明它表示什么状态或数据、由什么流程更新、被哪些逻辑消费、过期或不一致时会造成什么问题。尤其包括响应式状态、缓存数据、请求 token、权限标记、表单模型、选中状态、当前项指针、待处理任务句柄、父子组件联动状态。只有含义在使用点完全明显的短生命周期局部临时变量可以跳过。
- 方法/函数应使用注释或 JSDoc 说明职责、调用场景、关键参数、副作用和结果影响。公开/导出函数、组合式函数/hooks、事件处理器、请求封装、适配器和非平凡 helper 即使命名清晰也应补充说明。只有名称和签名已完整表达意图的极小局部 helper 或直接 wrapper 可以跳过。
- 关键分支应说明条件存在的原因，以及命中或未命中时对业务行为、状态一致性、权限、校验、兼容性、竞态保护、脏数据保护、异常输入处理或兜底行为的影响。
- 异步请求、并发控制、防抖/节流、缓存失效、状态重置、弹窗生命周期、组件卸载清理、存储清理、事件监听注册/移除，必须在关键判断或清理点说明处理意图。
- 涉及业务规则、权限控制、表单校验、后端字段映射、默认值、弹窗生命周期、父子组件联动、浏览器插件边界、第三方组件适配时，应说明约束和设计原因，而不是只描述实现细节。
- 给已有文件补注释时，必须保持行为和格式不变，同步更新附近过期注释，删除误导性或重复性说明；除非文件定义协议、状态机、集成边界或复杂工作流，否则避免宽泛的文件级叙述。
- 注释专项改动结束前，必须重新扫描已触达文件，检查是否仍有未注释的常量、重要状态、导出方法、关键分支、异步清理或过期注释；补齐遗漏的意图注释，或明确保留无需注释的明显代码。

## Element Plus 表单

- 只要新增或修改基于 `el-form` 的表单，默认使用 `FormInstance + FormRules` 进行校验。
- 不要使用 `if (!value) ElMessage.warning(...)` 这类临时必填判断作为主要校验方式。
- 提交时统一先执行 `await formRef.value?.validate()`；校验失败时立即返回。
- 需要校验的 `ElFormItem` 必须声明 `prop`，并在 `ElForm` 上绑定对应的 `:rules`。
- 清空、关闭弹窗或重置表单时，要同步调用 `resetFields()` 或 `clearValidate()`，避免残留校验状态。
- 只有在跨字段、异步或业务联动校验无法通过基础规则表达时，才使用自定义 `validator`。

## 弹窗约定

- 业务弹窗默认统一使用 `AppDialog` 组件实现；除非场景确有必要且说明原因，否则不要直接使用原生 `ElDialog`。
- 弹窗统一对外暴露 `toggleDialog(open, payload?)` 方法；父组件只通过弹窗实例调用该方法打开弹窗，不直接修改弹窗内部业务状态。
- 弹窗初始化逻辑必须放在弹窗内部，并在 `toggleDialog(true, payload)` 中直接执行；不要使用 `watch(visible)` 承担初始化、回填或重置逻辑。
- `handleCancel` 主要用于设置 `visible = false`，执行关闭动作。
- `handleClose` 主要用于清理副作用，包括重置表单、清空上下文数据、清理校验状态、使旧请求失效、重置加载状态、清空缓存等。
- `toggleDialog(false)`、取消按钮关闭、右上角关闭、提交成功后的关闭，都应复用同一套关闭流程，保证副作用清理一致。
- 弹窗内如包含 `el-form`，默认统一使用 `formRef + FormRules`，提交前先校验，关闭时执行 `resetFields()` 或 `clearValidate()`。
- 弹窗关闭或重新初始化后，过期请求的返回结果不得再回填到当前界面。
