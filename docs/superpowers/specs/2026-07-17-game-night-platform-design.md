# 今晚开局游戏平台设计规范

> 日期：2026-07-17
>
> 状态：已确认
>
> 目标：从零设计一个面向中国大陆用户、移动端优先、支持多人在线房间和持续扩展游戏的平台。

## 1. 背景与边界

本项目服务于异地朋友在线组局，现场同桌聚会是兼容场景，不是首要架构约束。平台首批支持骰子系列、三关定胜负和德州扑克，用户可以留在同一个持续房间中切换游戏。

参考目录仅用于理解玩法与发现体验问题：

- `.tmp` 包含摇骰子、转盘、抽牌、挖雷和指尖随机等同屏聚会工具。
- `E:/WorkPro/Project_20260305/nuxt-games` 包含三关、德州、房间、聊天、观战和复盘尝试。
- 新平台不复用上述目录的代码、架构、技术栈、资源或页面结构。

本设计不是 MVP 缩减方案。后续阶段只表达依赖顺序，不代表从正式范围删除能力。

## 2. 产品定位

### 2.1 核心场景

- 用户主要通过移动 Web/PWA 在线与异地朋友游玩。
- 房主创建持续房间，通过链接、二维码或房间码邀请其他用户。
- 平台同时提供私密房间与公开大厅。
- 语音由微信、QQ、Discord 等外部工具承担；平台提供文字、表情和快捷反应。
- 平台支持聚会游戏中的抽象惩罚单位，但不记录酒类、容量、毫升数或现实饮用行为；客户端可以将惩罚解释为饮料、积分或其他非酒精替代。主题层不得把规则层的抽象单位扩展为现实饮酒追踪。

### 2.2 商业边界

- 平台免费且无广告。
- 不建设支付、会员、虚拟商品或广告 SDK。
- 德州扑克筹码仅为当前房间内临时记分单位，不可购买、赠送、兑换、跨房间使用或提现。

### 2.3 正式目标

- 新用户从邀请链接进入目标房间的流程短、连续、可恢复。
- 公开大厅支持发现、筛选、准入、举报、封禁和房间状态展示。
- 新游戏通过稳定 SDK 接入，不修改平台房间分支或硬编码游戏联合类型。
- 所有游戏由服务端裁决，支持断线恢复、观战和规则约束下的复盘。
- 游戏界面专注、操作完整且简便，按风险减少误操作。
- 主题与规则解耦，可持续替换视觉资产、音效、触觉和动效。

### 2.4 非目标

- 首期不建设好友关系、私聊、群组、动态或长期社群。
- 首期不建设平台内语音或视频房间。
- 首期不建设陌生人自动匹配池；陌生玩家通过公开大厅进入房间。
- 不支持真实货币、可交易资产或任何提现路径。
- 不从第一天采用 Kubernetes 或大量微服务。

## 3. 用户身份

### 3.1 设备凭证

- 首次访问由服务端签发高熵设备凭证，不采集 Canvas、字体、硬件或网络特征形成浏览器指纹。
- 凭证通过 `Secure`、`HttpOnly`、`SameSite` Cookie 保存；客户端脚本不能读取长期秘密。
- 新设备身份必须设置唯一用户名后才能进入平台。
- 用户名用于公开展示和识别；后台真实姓名为管理员可见字段，不向其他玩家公开。
- 真实姓名的读取、修改和导出必须进入审计日志，并受后台 RBAC 控制。

### 3.2 恢复码

- 创建身份时展示一次恢复码，服务端只保存强哈希。
- 恢复成功后轮换恢复码；用户可选择撤销旧设备凭证。
- 后台可以执行人工重置，但不能查看原恢复码。
- 丢失凭证且无恢复码时只能通过后台人工流程处理，禁止依赖设备指纹自动匹配。

### 3.3 安全控制

- 写操作使用 CSRF 防护、会话轮换、速率限制和审计事件。
- 后台账号必须使用独立认证域、RBAC 和多因素认证。
- 恢复、用户名占用、举报和公开房加入接口必须具有设备、IP 与账号多维限流。

## 5. 持续房间模型

### 5.1 PartyRoom

`PartyRoom` 代表一群持续在一起的用户，而不是某一款游戏。

核心字段：

- `roomId`、`roomCode`
- `visibility`: `public | private`
- `hostPlayerId`
- `members` 与稳定 `seatIndex`
- `participantAdmission`: `open | approval | closed`
- `spectatorAdmission`: `open | approval | closed`
- `waitingMembers`
- `roomVersion` 与 `membershipVersion`
- `activeSessionId`
- `themeSelection`
- 房间聊天、快捷反应和治理状态

房间成员角色为：

- `participant`：拥有参赛座位。
- `spectator`：只能观看观众投影。
- `waiting`：等待房主批准成为参赛者。

### 5.2 房主与掉线

- 房主掉线时，对局继续运行，服务端规则不依赖房主在线。
- 房主在宽限期内重连时恢复房主身份。
- 宽限期结束后，房主身份转移给在线时间最长且未被限制的参赛者。
- 原房主之后重连时以普通参赛者身份返回，除非新房主主动交还。
- 房主转移必须产生房间审计事件。
- 没有合格继任者时，进行中的对局继续由服务端运行，但参赛许可保持关闭；对局结束后房间进入关闭流程。
- 没有活跃会话的空房间在可配置宽限期后过期，过期和显式解散都必须产生审计事件。

### 5.3 进房许可生命周期

1. 赛前大厅：房主可选择直接开放、需批准或关闭。
2. 开始游戏：冻结参赛名单和 `seatIndex`，参赛许可自动锁定。
3. 对局进行中：原参赛者可重连；新用户只能观战、候场或看到“暂不可加入”。
4. 单局结束：返回赛后大厅，现有成员保留，新增参赛许可默认继续关闭。
5. 房主重新开放：房主选择直接开放或需批准，优先处理候场玩家，再准备下一局。

约束：

- 公开房只代表可在大厅发现，不代表允许中途入座。
- 对局进行中 `participantAdmission` 的有效值强制为 `closed`。
- 原参赛者重连不受新增许可影响。
- 新玩家只加入 `PartyRoom`，不写入上一局 `GameSession`。
- 批准和直接加入必须原子校验房主、`roomVersion`、`membershipVersion` 与剩余容量。
- `spectator/waiting -> participant` 使用独立命令并产生审计事件。

### 5.4 成员移除、封禁与解散

- `GameSession` 中冻结的参赛者和座位快照用于规则与历史，不等同于持续有效的访问权限。
- 房主踢人或管理员封禁立即撤销该用户对房间和会话的新命令、订阅与重连权限。
- 被移除者若属于当前参赛者，运行时向游戏引擎发送平台级 `ParticipantRevoked` 系统命令；游戏规则规范必须定义超时、弃权、自动操作或取消等确定性处理，且该座位在本局结束前不得交给新玩家。
- 移除不能改写已经提交的事件；原因、操作者、规则处理和结果影响必须进入审计与复盘摘要。
- 解散进行中的房间属于高风险操作，确认后将当前会话标记为 `cancelled`，不生成正常胜负结算，撤销所有连接权限并进入关闭流程。
- 管理员紧急封禁或解散优先于房主配置；客户端收到稳定原因码后退出游戏视图，不得继续使用离线草稿提交命令。
- 房主踢人或拉黑不撤销该用户作为历史参赛者已经取得的默认复盘权限；管理员封禁可以在封禁期内暂停其全部复盘访问，且必须记录原因、期限和审计事件。

### 5.5 公开大厅状态

公开房卡片根据状态展示不同主动作：

- 赛前开放：直接加入。
- 赛前需批准：申请加入。
- 对局进行且允许观战：进入观战或加入候场。
- 对局进行且禁止观战：对局中。
- 赛后关闭：等待房主开放。
- 已满员：仅观战或不可加入。

## 6. 游戏目录

### 6.1 骰子系列

骰子作为游戏系列接入，首批模式包括：

- 吹牛骰子：每位玩家看到自己的私密骰子，按顺序叫点或开骰，由服务端揭示和结算。
- 789：服务端管理轮次、骰点、方向、公共状态和结算规则。
- 喜相逢：服务端管理多骰比较、同牌/同大/同小重摇、靶子换位和结算。

模式共享骰子领域与主题资产，但拥有独立配置、动作、状态机和规则版本。

### 6.2 三关定胜负

- 2-9 人，推荐 2-6 人。
- 每人六张牌，第一关选一张、第二关选两张、第三关使用剩余三张。
- 服务端管理洗牌、私密手牌、分阶段提交、每关比较和最终积分。
- 客户端只接收自己的手牌、他人的提交状态和公共结算结果。

### 6.3 德州扑克

- 2-9 人。
- 服务端管理座位、庄家、盲注、下注轮、边池、摊牌和结算。
- 筹码在房间内创建，离开房间或房间结束后不形成可交易资产。
- 未摊牌底牌默认不在观众或复盘中公开。

### 6.4 独立规则规范

平台总体规范不替代具体玩法规则。进入对应引擎实施计划前，吹牛骰子、789、喜相逢、三关定胜负和德州扑克必须分别完成并审查独立规则规范，至少冻结：

- 地区或房间规则变体、人数与初始配置
- 回合、合法动作、计时、断线和 `ParticipantRevoked` 处理
- 比较顺序、计分、平局、结束与异常取消
- 随机性、私密信息、摊牌和复盘揭示边界
- 可执行示例、边界用例和属性测试不变量

## 7. 游戏体验与页面布局

### 7.1 共同桌面

- 房间和游戏均以共同桌面组织玩家，不使用割裂的头像列表和管理面板作为主要体验。
- 稳定的是逻辑座位顺序，不是固定圆桌形状。
- 大厅中成员变化可以平滑重排；对局开始后冻结 `seatIndex`。
- 断线、重连和横竖屏切换只改变几何投影，不改变相对顺序。

桌形规则：

- 2-4 人：紧凑圆形或短椭圆。
- 5-6 人：根据屏幕方向拉伸椭圆。
- 7-9 人：长椭圆或圆角桌。
- 骰子、三关和德州可以在 Manifest 中声明桌形偏好与中心安全区。
- 头像保持最低触控尺寸，布局器按完整座位包围盒与操作托盘安全区计算位置。

### 7.2 横竖屏

- 竖屏是完整主体验，核心设计基线为常见长方形移动视口。
- 横屏是同一桌面的重排，不要求用户横屏才能完成游戏。
- 旋转不重建连接、房间订阅、游戏状态或计时器。
- 选牌、下注额、叫点草稿和托盘状态在旋转后保留。
- CSS 响应式布局是基础；仅在浏览器支持时请求系统方向锁定，不能依赖 iOS 浏览器强制旋转。

### 7.3 滑动操作托盘

操作区域从“我的座位”延伸，不使用与游戏画面割裂的白色表单面板。

- 收起态约占竖屏 10%：等待其他玩家时显示私密状态摘要。
- 紧凑态约占竖屏 18%-24%：轮到自己时显示高频合法动作。
- 展开态最多约占竖屏 41%：用户主动上滑后显示精细调整和低频能力。
- 手柄支持拖动吸附和轻点切换。
- 轮到玩家时最多自动提升到紧凑态，不强制完全展开。
- 提交后回落到收起态。
- 横屏使用低矮下沿托盘，不能固定占据半个屏幕宽度。

### 7.4 游戏化操作

- 骰子使用骰盅托盘、叫点拨盘和开骰动作。
- 三关使用从座位展开的手牌扇面和桌面出牌反馈。
- 德州使用筹码轨道和筹码进入底池的动作反馈。
- 平台只提供座位层、HUD、安全区、托盘容器、主题和 dispatch 端口，不输出通用业务表单。

### 7.5 专注与防误操作

- 游戏中不展示大厅导航；设置、帮助、举报和离开进入二级菜单。
- 客户端只渲染当前状态允许的动作，服务端再次校验。
- 高频且预期的动作单击提交，不滥用确认框。
- 不可逆且高影响动作按风险使用长按、滑动确认或二次确认。
- 危险按钮与主按钮保持物理间距，位置不因倒计时或网络状态突然变化。
- 提交中的按钮原地锁定并显示 pending，禁止重复提交。
- 每个动作携带唯一 `actionId`，服务端保证幂等结果。
- 无上下文价值的非法动作隐藏；需要解释时禁用并显示原因。

## 8. 主题系统

### 8.1 三层主题

1. 平台基础令牌：字体、触控尺寸、安全区、层级、语义状态、对比度、焦点和减少动效。
2. 游戏主题包：桌面材质、色板、骰盅、牌背、筹码、音效、触觉、粒子和转场。
3. 运行时偏好：浅色/深色、高对比度、减少动画、静音与后台启用版本。

### 8.2 版本与回退

- 每个主题包具有 `themeId`、版本、兼容游戏版本、资源清单和完整性哈希。
- 游戏 Manifest 声明默认主题、可用变体与回退主题。
- 主题资源加载失败时必须回退到内置安全主题，核心操作不可失效。
- 进行中的会话固定主题资源版本，防止局中资产变化。

### 8.3 不可覆盖边界

主题允许改变表现层，但不得改变：

- 动作含义和顺序
- 危险等级
- 最小触控尺寸
- 服务端规则
- 私密信息边界
- 高对比度与减少动效等平台安全偏好

## 9. 观战与复盘

### 9.1 观战

- 公开房默认可观战，私密房由房主配置。
- 观众不占参赛座位，可以进入候场席。
- 观众只接收 `SpectatorProjection`，不得收到当前不应公开的骰子或手牌。
- 外部语音不由平台录制或保存。

### 9.2 复盘

- 服务端保存单局事件和周期快照。
- 复盘由 `ReplayProjection` 生成，不直接把数据库权威状态返回客户端。
- 游戏定义揭示策略；未摊牌底牌可以永久隐藏。
- 复盘资源授权与字段投影分开执行；通过资源授权后，仍只能收到规则允许的 `ReplayProjection`。
- 默认只有该局参赛者可以查看复盘。房主可将权限扩大为该局结束时的房间成员快照；只有公开房可以显式配置为所有已登录用户可见，私密房不得公开复盘。
- 观战、持有链接或知道 `replayId` 本身不授予访问权。管理员访问必须具有独立 RBAC 权限并写入审计日志。
- 原始事件、权威快照和完整归档永不直接提供给客户端，也不得为其签发下载 URL。
- API 完成资源授权后，由服务端生成允许的 `ReplayProjection`。若使用短期签名 URL，它只能指向按 `viewer + policy` 物化且不含永久隐藏字段的安全投影产物；否则由服务端代理返回投影。
- 完整复盘默认保留 30 天，后台可配置。
- 保留期结束后可删除或归档完整事件，长期保留结果摘要和治理审计。

## 10. 服务端架构

### 10.1 部署单元

- `api`：ConnectRPC、身份、房间、大厅、复盘和后台接口。
- `realtime`：WebSocket 网关、会话归属和游戏会话运行时。
- `worker`：快照、归档、清理、审核和异步任务。
- `web/admin`：由 Nginx 提供静态资源。

平台采用模块化内核。身份、资料、房间、大厅、治理、主题、复盘和后台是内部模块，不在第一阶段拆成微服务。

### 10.2 PartyRoom 与 GameSession

`PartyRoom` 管理“这群人”；`GameSession` 管理“这一局”。同一房间连续创建多个游戏会话。

`GameSession` 核心字段：

- `sessionId`、`gameId`
- `engineVersion`、`protocolVersion`、`clientVersion`
- `stateVersion`
- `ownershipEpoch`
- 冻结参赛者与座位快照
- 权威游戏状态
- 回合截止时间
- 事件批次与周期快照
- 结束状态和复盘策略

### 10.3 命令处理

1. 客户端发送 `actionId`、`sessionId`、`expectedVersion`、`gameId`、消息类型和 payload。
2. 实时网关鉴权并路由到会话归属实例。
3. 运行时以 `sessionId + actorId + actionId` 查询 PostgreSQL 持久动作回执；相同请求直接返回原回执，不重新执行，复用同一 ID 但请求摘要不同则拒绝。
4. 没有回执时，引擎校验权限、版本、回合、动作和参数，生成领域事件与新状态。
5. 同一 PostgreSQL 事务写入事件批次、会话版本、outbox 和动作回执；唯一约束处理并发重复提交。
6. 动作回执保存稳定结果码、提交后的 `stateVersion` 和安全结果摘要；重试时按当前身份重新同步可见投影，不持久化其他玩家秘密。
7. 投影器为各玩家、观众和复盘生成不同视图。
8. 网关推送新版本和动作结果，客户端解除 pending。

### 10.4 会话归属与计时器

- 一个活跃 `GameSession` 同时只归属一个运行实例。
- Redis 保存带 TTL 的会话租约和实例地址。
- Redis 租约只负责存活与路由；每次成功取得新归属时，PostgreSQL 中的 `ownershipEpoch` 单调递增并作为权威 fencing token。
- 所有会话状态写事务都必须携带当前 epoch，并以 `ownership_epoch = ?` 为提交条件，包括玩家命令、计时器、`ParticipantRevoked`、解散和恢复；已经失去归属的旧实例即使仍在运行，也不能提交事件、快照或动作回执。
- 跨实例命令通过内部 ConnectRPC 路由到归属实例，不使用易丢失的 Pub/Sub 传递权威命令。
- Redis Pub/Sub 只用于非权威通知和热点刷新。
- 计时器截止时间写入权威状态；内存计时器只负责唤醒，实例恢复时重新调度。

### 10.5 断线恢复

1. 设备凭证恢复身份与最近房间。
2. 客户端报告最后确认的 `stateVersion`。
3. 差距小时补发可见事件，差距大时返回最新可见快照。
4. 重新订阅房间与会话。
5. 倒计时基于服务端截止时间计算，不使用本地重置计时。

## 11. 可见性投影

同一权威状态必须显式生成三类接收者投影：

- `PlayerProjection(viewerPlayerId)`：面向一个具体玩家的完整视图，包含该玩家自己的秘密、可执行动作、公共桌面，以及只含规则允许字段的 `opponents[]` 公共子视图。
- `SpectatorProjection(viewerId)`：面向观众的视图，不包含当前不应公开的信息；观众身份只用于资源授权和个性化公共状态。
- `ReplayProjection(viewerId, policy)`：资源授权通过后，按游戏揭示与保留策略生成。

`opponents[]` 不是独立推送给其他玩家的第四类投影，也不允许客户端用同一个完整玩家对象自行隐藏秘密字段。`ServerGameModule.project(state, viewer)` 根据 viewer 的玩家或观众角色生成前两类投影，复盘只通过 `projectReplay` 生成。

禁止先把权威状态广播到浏览器，再依赖前端隐藏字段。投影泄密测试是每款游戏的强制 CI 门禁。

## 12. Game SDK

### 12.1 GameManifest

Manifest 声明：

- 游戏 ID、引擎版本、协议版本、客户端模块版本
- 最小/最大人数和座位能力
- 回合制/同时提交/计时器能力
- 观战、复盘与揭示策略
- 桌形、方向和托盘偏好
- 默认主题、主题变体和回退主题

### 12.2 ServerGameModule

服务端模块提供：

```text
create(context, config)
handleCommand(state, command)
handleTimer(state, timer)
project(state, viewer)
projectReplay(events, viewer, policy)
migrate(snapshot, fromVersion, toVersion)
```

引擎要求：

- 时钟、随机数和 ID 由运行时注入。
- 不执行网络、数据库、Redis、文件或环境变量 IO。
- 相同初始状态、输入、时间和随机种子必须产生相同事件。

### 12.3 ClientGameModule

客户端模块只接收 `GameView` 和 `AllowedActions`，通过 `dispatch` 发送命令。它可以实现：

- `TableSurface`
- `SeatAdornment`
- `ActionDock`
- `ReplayScene`
- 游戏主题与动作反馈

客户端不得导入服务端权威状态类型或直接管理 WebSocket。

### 12.4 注册

- 每款游戏在自己的目录维护 Manifest 和协议。
- 构建阶段生成客户端与服务端注册表。
- 平台注册表不维护所有游戏的硬编码联合类型或 `switch` 分支。
- 运行时只加载构建时允许且通过签名/版本校验的游戏模块。
- 注册表按 `gameId + engineVersion + protocolVersion + clientVersion` 索引；只要仍有可恢复会话引用，服务端就必须保留对应引擎和编解码器。
- 客户端游戏模块使用内容哈希的版本化静态地址，平台壳在进入或恢复会话时加载该会话指定版本，不依赖当前默认版本。

## 13. 跨语言协议

### 13.1 Protocol Buffers + Buf

- `.proto` 是 Go 与 TypeScript 的协议来源。
- Buf 执行格式、Lint、代码生成和破坏性变更检查。
- HTTP API 使用 ConnectRPC。
- WebSocket 使用 Protobuf 二进制帧。
- 平台定义通用实时信封，不把所有游戏写入平台级 `oneof`。
- 每款游戏拥有独立 proto 包和 payload 编解码器，由构建注册表连接。

通用实时信封至少包含：

- `action_id`
- `session_id`
- `expected_version`
- `game_id`
- `schema_version`
- `message_type`
- `payload`

### 13.2 兼容性

- 运行中的会话固定引擎和协议版本。
- 新版本不能热替换进行中的会话。
- Buf breaking check 阻止不兼容字段删除或复用。
- 快照迁移必须逐版本、可测试、可回退。
- 发布产物可以同时包含多个引擎、协议编解码器和客户端模块版本；只有当数据库中不存在引用且恢复/复盘保留窗口结束后才允许清理旧版本。
- 精确版本暂时不可用时会话进入 `suspended`，禁止自动迁移、静默终止或使用新版本继续；恢复对应产物后再按原版本加载。

## 16. 数据存储

### 16.1 PostgreSQL

持久权威数据包括：

- 设备身份、用户名、恢复码哈希和设备撤销
- 房间、成员、角色、许可和审计
- 游戏会话版本、事件批次和周期快照
- 会话归属 epoch、动作请求摘要与持久动作回执
- 举报、封禁、后台操作和真实姓名访问审计
- 复盘索引、保留期和结果摘要
- 主题与游戏启用版本

每个会话的 `stateVersion` 单调递增。事件批次、版本和 outbox 在同一事务提交。

### 16.2 Redis

Redis 不是持久权威，仅保存：

- 在线状态
- 会话归属租约
- 持久动作回执的短期热点缓存
- 热点只读缓存
- 跨实例非权威通知

动作回执缓存只能在 PostgreSQL 事务提交后写入，并保留请求摘要校验信息；缓存命中仍须校验当前身份，响应按当前 viewer 重新生成可见投影。

### 16.3 对象存储

使用 OSS/COS/S3 兼容服务保存：

- 版本化主题资产
- 音效和游戏资源
- 归档复盘
- 导出与审核附件

对象路径包含内容哈希和版本。主题等公开资产可以使用受控公开地址；复盘、导出和审核对象必须执行各自资源授权，且不得绕过第 9.2 节的安全投影约束。

## 17. 单机生产部署

### 17.1 基线

- 单台 `4C8G+` Linux 应用服务器。
- Docker Compose 或 1Panel 管理容器。
- PostgreSQL 和 Redis 使用现有外部服务。
- 对象存储使用外部 OSS/COS/S3 服务。
- Nginx 负责 TLS、静态文件、ConnectRPC 和 WebSocket 反向代理。

容器：

```text
nginx
web-static
admin-static
api
realtime
worker
otel-collector（可按资源情况外置）
```

### 17.2 发布

- CI 构建并扫描镜像，生产机只拉取和启动。
- 发布清单列出新实例可加载的全部引擎和协议版本；部署前校验其覆盖数据库内所有可恢复会话引用。
- 版本化客户端游戏模块发布到不可变静态路径，旧文件在引用与保留窗口结束前不得删除。
- 蓝绿应用栈临时并存，健康检查通过后切换 Nginx upstream。
- 实时实例进入 draining 后停止接收新连接，通知客户端重连，并等待旧会话排空。
- 数据库迁移采用向后兼容的 expand/contract 流程，禁止与旧实例不兼容的单步迁移。

### 17.3 单机风险

- 应用服务器故障会造成整体服务中断，即使外部 PostgreSQL/Redis 仍可用。
- 单机维护期间无法提供真正高可用。
- 容量和网络上限必须通过 k6 验证。
- 所有容器和配置必须保持无状态、可迁移，为后续双节点或容器服务升级做准备。

## 18. 错误处理

### 18.1 协议错误

- 所有错误使用稳定错误码和安全用户文案。
- 不向客户端返回堆栈、数据库错误或私密状态。
- 版本冲突返回当前服务端版本，并指示客户端补事件或获取快照。
- 非法动作返回禁用原因，但不能泄露其他玩家秘密。

### 18.2 客户端行为

- 提交后原按钮进入 pending，禁止重复点击。
- 网络中断时保留本地操作草稿，但不假定命令成功。
- 重连后以服务端确认的 `stateVersion` 为准。
- 不自动重放可能产生副作用且未确认幂等状态的动作。

### 18.3 服务端故障

- PostgreSQL 提交失败时不广播新版本。
- Redis 故障时停止取得新租约；已持有会话根据安全策略降级或暂停，不能产生双主。
- 对象存储故障不阻断核心对局；主题回退到内置安全主题，归档任务进入重试队列。

## 19. 测试策略

### 19.1 Go

- 标准单元测试覆盖规则、房间许可、投影和错误码。
- 原生 fuzz 覆盖命令序列、牌型、边池、骰子比较和协议解析。
- `go test -race` 覆盖会话运行时、租约和连接管理。
- Testcontainers 覆盖 PostgreSQL、Redis、迁移、outbox 和恢复。
- 并发与故障测试覆盖持久动作回执、ID 复用冲突、Redis 丢失和 ownership epoch fencing。
- 引擎确定性测试验证相同输入重放产生相同事件。

### 19.2 TypeScript/Vue

- Vitest 覆盖状态适配、主题、布局器和动作组件。
- Playwright 覆盖身份、邀请、公开房、重连、观战、复盘和房主赛后开放。
- 固定视觉视口至少覆盖 `390x844`、`844x390` 和较小安卓设备。
- axe-core 覆盖焦点、对比度、语义和键盘访问。

### 19.3 Game SDK 契约门禁

每款游戏必须通过：

- Manifest 与协议校验
- 确定性和随机种子重放
- 非法命令与动作幂等
- 玩家自身/对手公共子视图、观众和复盘投影泄密扫描
- 事件重放与快照恢复
- 引擎/协议/快照迁移
- 旧版服务端引擎与客户端模块保留、精确版本恢复和缺失版本暂停
- 计时器恢复
- 横竖屏和安全区
- 主题加载失败与回退
- 复盘揭示策略

### 19.4 压测目标

- 1,000 同时在线玩家。
- 多个热点公开房和批量观众订阅。
- WebSocket 连接、心跳、断线重连和蓝绿排空。
- 国内同区域网络下，服务端动作确认 p95 目标小于 250ms，不含表现动画。
- 网络恢复后，客户端恢复到最新可见状态的目标小于 5 秒。
- 测试结果决定单机真实容量，不能只按连接数估算。

## 20. 可观测性与治理

### 20.1 指标

- 当前连接、房间、活跃会话、观众和候场数量
- 命令吞吐、确认延迟、拒绝率、版本冲突和重复动作
- 会话租约续期、转移和恢复次数
- 事件提交、快照、outbox 和 Worker 延迟
- PostgreSQL、Redis、对象存储和 Nginx 指标

### 20.2 日志与追踪

- 结构化日志使用 trace ID、room ID、session ID、action ID 和 stateVersion。
- 不记录恢复码、设备秘密、完整真实姓名或未脱敏游戏秘密。
- OpenTelemetry 串联 HTTP、WebSocket 命令、引擎执行、数据库提交和推送。

### 20.3 治理

- 公开房支持房主踢人、举报、拉黑、昵称审核和异常退出处理。
- 举报证据采用最小必要原则保存。
- 封禁和后台修改必须产生不可抵赖审计记录。

## 21. 实施阶段

以下阶段表达依赖顺序，不是 MVP 删减：

1. 工具链与仓库：Go、pnpm、Buf、代码生成、依赖边界和 CI。
2. 身份与后台：设备凭证、用户名、恢复码、真实姓名和审计。
3. 房间与大厅：持续房间、公开大厅、许可、成员角色、聊天和治理。
4. 实时运行时：WebSocket、会话租约、命令、事件、快照、重连和投影。
5. Game SDK 与主题：Manifest、引擎接口、客户端端口、测试套件和主题运行时。
6. 骰子系列：先完成吹牛、789、喜相逢的独立规则规范，再分别实现与验证。
7. 三关定胜负：先完成独立规则规范，再实现与验证。
8. 德州扑克：先完成独立规则规范，再实现与验证。
9. 观战、复盘、保留策略和后台运营。
10. 压测、安全、故障演练、蓝绿发布和生产上线。

## 22. 验收标准

- 用户可通过设备凭证创建身份、设置用户名并使用恢复码恢复。
- 邀请深链在身份流程后自动返回目标房间。
- 私密房和公开房均可创建、加入、观战、候场和治理。
- 同一房间可连续切换三款游戏而不重建成员关系。
- 对局中新增参赛席锁定，原玩家可重连；赛后房主可重新开放。
- 踢人、封禁和解散会立即撤销访问权限，并按游戏规则确定性处理冻结座位。
- 三款游戏均由服务端权威裁决并通过 Game SDK 契约门禁。
- 玩家自身、对手公共子视图、观众和复盘投影不存在私密信息泄漏，复盘资源授权不能被链接或对象地址绕过。
- 同一动作跨进程重试不会重复产生领域事件，失去租约的旧实例不能提交新版本。
- 发布和故障恢复可以按会话记录加载精确的服务端引擎、协议和客户端模块版本。
- 竖屏完整可玩，横屏自适应，旋转不重置状态。
- 操作托盘可收起、紧凑和展开，危险动作具有适当防误触。
- 主题可版本化切换和回退，不改变规则或安全语义。
- 单机部署可从外部 PostgreSQL/Redis 和事件快照恢复。
- 1,000 同时在线压测、WebSocket 重连、蓝绿排空和故障恢复达到目标。

## 23. 已知风险

- 单台应用服务器仍是可用性单点；后续应升级为至少双节点。
- Go 与 TypeScript 双语言增加协议生成和构建复杂度，必须依赖 Buf 和 CI 防止漂移。
- 德州扑克边池、全下和复盘揭示规则复杂，必须先完成独立规则规范与属性测试。
- 公开大厅带来治理、滥用和数据合规成本，不能在实现时降级为无审核公开列表。
- 浏览器横屏锁定能力不一致，必须以响应式布局而不是强制方向为基础。
- 主题资产可能体积较大，需要版本化、缓存、完整性校验和安全回退。

## 24. 决策摘要

- 线上异地联机优先，现场聚会兼容。
- 中国大陆移动 Web/PWA 首发。
- 服务端设备凭证 + 唯一用户名 + 恢复码，不使用浏览器指纹。
- 无好友系统、无内置语音、现实饮酒不追踪、惩罚单位抽象化、无广告和支付。
- 私密房 + 公开大厅，持续房间可切换游戏。
- 实时观战 + 规则约束复盘。
- 模块化平台内核 + 独立实时网关 + Game Session Runtime。
- Vue 3/Vite/TypeScript 前端，Go 后端。
- Protocol Buffers + Buf + ConnectRPC + Protobuf WebSocket。
- PostgreSQL 权威持久化，Redis 非权威协调，OSS/COS/S3 保存资产和归档。
- 单台 Docker Compose/1Panel 应用服务器作为首套生产部署，保持向双节点迁移能力。
