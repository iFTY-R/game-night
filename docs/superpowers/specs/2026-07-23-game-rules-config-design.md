# 三款骰子游戏房主规则配置设计

> 日期：2026-07-23
>
> 状态：设计已获确认，待编写实现计划
>
> 适用游戏：`liars-dice`、`dice-789`、`meet-by-chance`

## 1. 目标与范围

为房间开局前增加完整的房主规则配置能力，移动端竖屏优先，同时支持横屏。配置必须覆盖现有游戏引擎已经定义的全部用户可调规则，规则抽屉要与共同游戏桌保持连续的视觉和交互关系，避免房间页、操作面板和游戏本体割裂。

本设计覆盖：

- 房间内当前游戏选择与规则草稿。
- 三款骰子游戏的完整规则配置、默认值、依赖联动和校验。
- 所有房间成员查看、仅房主编辑的权限模型。
- 房间内规则草稿同步、修订冲突和服务端权威校验。
- 房主个人规则方案的保存、应用、覆盖和删除。
- 开局时的配置冻结，以及游戏桌和复盘中的规则快照展示。
- 移动端竖屏、横屏和误操作防护。

不在本次实现范围内：

- “三关定胜负”和德州扑克的规则配置；它们尚未注册为可玩的游戏模块。
- 修改三款游戏本身已经确认的玩法语义。
- 将现实酒类、容量或饮用行为写入服务端；惩罚继续使用抽象单位。

## 2. 已确认的产品决策

### 2.1 规则入口

采用“桌面上的底部抽屉”方案：

- 竖屏时，房间桌面只展示当前游戏、规则摘要和“查看规则”入口。
- 规则抽屉支持中等高度和近全屏两档，内部独立滚动，不永久占据房间一半以上空间。
- 横屏时，同一规则抽屉改为右侧面板，左侧保留玩家桌面和围桌关系。
- 抽屉底部保留固定操作区，避免滚动后找不到“完成”或“恢复默认”。

### 2.2 权限与可见性

- 所有房间成员都能打开并查看当前规则。
- 只有房主能编辑、应用个人方案、保存方案和恢复默认。
- 非房主使用只读展示，不显示看起来可点击但实际无效的禁用开关。
- 房主修改后的规范化规则实时同步给所有成员。
- 个人方案只属于房主本人，其他成员看不到方案名称和方案列表，只能看到已应用到房间的规则。

### 2.3 草稿、切桌与冻结

- 草稿按 `roomId + gameId` 独立保存；房主切换游戏后再切回来，继续使用该游戏草稿。
- 房间解散后清理房间草稿。
- 个人方案跨房间保留，支持命名、覆盖、另存、删除和最近使用。
- “完成”只关闭规则抽屉，不触发开局。
- 开始游戏按钮位于房间桌面主操作区，与规则编辑分离。
- 点击开始后显示全员可见的 3 秒倒计时，房主可取消。
- 倒计时是服务端持久化的待开局状态，包含服务端截止时间、取消令牌、房间/成员/房主权属版本；重连按服务端截止时间计算剩余时间。倒计时期间规则、参赛名单、房主权属或房间状态发生变化时，服务端自动使令牌失效并取消倒计时。
- 服务端成功创建 `GameSession` 后，才正式冻结规则、参与名单和座位；开局失败不改变房间草稿。
- 游戏桌和复盘读取同一份冻结规则快照。对局开始后规则只能查看，不能修改。

## 3. 系统边界与数据归属

平台层提供统一能力：

- 规则底部抽屉、横屏右侧面板和规则摘要。
- 房主/成员权限、房间草稿、修订号、同步状态和冲突处理。
- 个人规则方案 CRUD 和最近使用记录。
- 开局时的版本校验、配置冻结、请求幂等和错误呈现。

游戏模块提供强类型规则适配器：

- 默认规则构造。
- 配置字段的标签、说明、控件类型和分组。
- 依赖联动和客户端预校验。
- 配置载荷的编码、解码、规范化和服务端校验。
- 面向房间摘要、游戏桌和复盘的规则展示。

平台复用现有 `GameEnvelope` 结构承载配置，不解释游戏字段：

- `gameId`
- `version.engine`
- `version.protocol`
- `version.client`
- `schemaVersion`
- `messageType`
- `payload`

其中 `schemaVersion` 是配置消息本身的 schema 版本；个人方案只保存完整 envelope，不再另造一层含义不同的 schema 字段。这样新增游戏不会要求修改房间核心模型，同时每款游戏可以保留自己的术语和控件。旧版方案读取时先按 envelope 版本选择适配器，再执行显式迁移；没有迁移器就标记不可用，禁止按默认值静默解码。

房间草稿需要至少包含：

- `roomId`
- `gameId`
- `configEnvelope`
- `revision`
- `updatedBy`
- `updatedAt`

个人方案需要至少包含：

- `presetId`
- `ownerUserId`
- `gameId`
- `name`
- `configEnvelope`
- `presetRevision`
- `createdAt`
- `updatedAt`
- `lastUsedAt`

方案更新、覆盖、删除和最近使用记录都以 `presetRevision` 为乐观并发版本；写请求携带 `expectedPresetRevision` 和请求 ID。

GameSession 的冻结规则快照还必须保留：

- `engineVersion`
- `protocolVersion`
- `clientVersion`
- 规范化 `configEnvelope`
- 配置载荷 hash（对规范化 envelope 的稳定字段序列化字节计算），用于复盘一致性和缓存校验
- 创建时的 `roomVersion`、`membershipVersion` 和 `ownershipEpoch`
- 冻结参赛者、座位快照和冻结时间

## 4. 三款游戏的配置清单

所有惩罚字段使用用户可理解的“惩罚单位”显示，服务端继续以整数 ticks 保存，`4 ticks = 1 unit`。

### 4.1 吹牛骰子 `liars-dice`

人数范围为 2-8 人；以下配置在会话创建时按实际参赛人数再次校验：

| 字段 | 默认值 | 可配置范围/行为 | 展示分组 |
| --- | --- | --- | --- |
| 每人骰子数 | 5 | 3-6 | 骰子与叫法 |
| 1 点万能 | 开启 | 开关 | 骰子与叫法 |
| 叫斋 | 开启 | 开关 | 骰子与叫法 |
| 飞斋 | 开启 | 仅叫斋开启时可用；叫斋关闭时自动关闭 | 骰子与叫法 |
| 首叫下限 | 当前参赛人数 | 1 至当前参赛人数 | 骰子与叫法 |
| 单轮输家惩罚 | 1 单位 | 0.5-2 单位，步长 0.25 | 惩罚与节奏 |
| 叫数时限 | 30 秒 | 关闭或 10-120 秒 | 惩罚与节奏 |

### 4.2 789 `dice-789`

人数范围为 2-12 人；以下配置在会话创建时校验容量乘法和步长关系：

| 字段 | 默认值 | 可配置范围/行为 | 展示分组 |
| --- | --- | --- | --- |
| 初始公共池 | 0.5 单位 | 0-16 单位 | 公共池 |
| 单层公共池上限 | 2 单位 | 0.25-16 单位 | 公共池 |
| 加注步长 | 0.25 单位 | 0.25-4 单位，必须整除单层上限 | 公共池 |
| 叠杯 | 关闭 | 开启后显示最大层数 | 公共池 |
| 最大层数 | 1 | 叠杯关闭时固定为 1；开启时 2-6，默认 3 | 公共池 |
| 普通对子反转 | 开启 | 开关 | 特殊对子 |
| 双 1 | 开启 | 可独立关闭 | 特殊对子 |
| 双 4 | 开启 | 可独立关闭 | 特殊对子 |
| 双 6 | 开启 | 可独立关闭 | 特殊对子 |
| 789 后续 | 可选 | 可选重摇/过牌、强制重摇、强制过牌 | 节奏 |
| 操作时限 | 30 秒 | 关闭或 10-120 秒 | 节奏 |
| 掉骰报告窗口 | 5 秒 | 3-15 秒 | 现场确认 |

现有 protobuf 字段仍为 `last_digit_match`；适配器对外统一映射为 `sumMatch: exact | last_digit`，默认 `exact`。固定两颗骰子时两种结果等价，因此不作为当前房主可见选项，避免提供没有实际玩法差异的控制；编码和复盘必须继续保留该字段，确保旧协议兼容。

789 的配置提交还必须明确执行以下约束：

- `initialPoolTicks <= layerCapacityTicks * maxLayers`。
- 叠杯关闭时 `maxLayers = 1`，总容量等于单层容量。
- 叠杯开启时 `maxLayers` 必须为 2-6。
- `layerCapacityTicks * maxLayers` 在乘法前检查溢出，超过 `uint32` 或 `384 ticks` 的规则总容量上限时拒绝。
- `addStepTicks` 必须整除 `layerCapacityTicks`，剩余容量不足一个步长时使用规则定义的填满行为。

## 5. 房间交互与同步流程

### 5.1 加载与展示

房间加载时返回当前游戏、各游戏草稿、草稿修订号和个人方案摘要。客户端根据 `gameId` 选择游戏适配器，解码当前草稿，生成房间摘要和规则抽屉。

房主切换游戏时，切换动作本身是房间级状态更新，携带 `expectedRoomVersion`、当前 `ownershipEpoch` 和请求 ID；服务端用同一套房主权限、版本冲突和幂等规则处理。所有成员同时看到新的桌面主题、最低人数要求和对应规则摘要。每款游戏草稿独立保留。

### 5.2 编辑与修订

房主修改规则时：

1. 客户端进行字段校验、依赖联动和规范化。
2. 发送 `UpdateRoomGameConfig`，携带 `roomId`、当前选中的 `gameId`、规范化配置 envelope、`expectedRevision`、`expectedRoomVersion`、`expectedMembershipVersion`、当前 `ownershipEpoch` 和请求 ID。
3. 服务端确认房主身份、房主权属 epoch、房间状态、当前选中的游戏、参赛人数、规则版本和游戏模块校验。
4. 校验通过后以事务方式保存草稿、递增修订号并广播标准化结果。
5. 其他成员只接受服务端返回的标准化配置，不使用自己的本地草稿覆盖房间状态。

开关更新可以立即提交；数字输入、步进器和滑块在操作结束时提交，避免每次按键都产生远端更新。

### 5.3 个人方案

房主可以从当前草稿保存新方案、覆盖已有方案、另存为新方案、恢复默认或删除方案。覆盖、删除和更新最近使用时间都携带 `expectedPresetRevision` 和请求 ID；修订不一致时返回最新方案，不覆盖新数据。应用个人方案等同于一次完整的房间规则更新，不直接修改历史对局。

个人方案按游戏分组；方案协议版本不兼容时显示为不可用并要求重新配置，禁止静默套用默认值。

### 5.4 开局与冻结

房主点击桌面上的开始游戏后，服务端创建待开局记录并广播 `pendingStartId`、`cancelToken` 和 3 秒截止时间；所有成员按该截止时间显示倒计时，房主可以取消。倒计时结束后，`StartGame` 携带 `pendingStartId`、`cancelToken`、请求 ID、当前游戏 ID、规则 envelope、规则修订号、`expectedRoomVersion`、`expectedMembershipVersion` 和 `ownershipEpoch`。

服务端依次确认：

- 请求者仍是房主。
- 房间仍处于允许开局的状态。
- 当前选中的 `gameId` 与配置 envelope 的 `gameId` 一致。
- 参赛人数满足游戏要求。
- 修订号没有过期。
- 房间版本、成员版本和房主权属 epoch 没有过期。
- 游戏模块能按实际参赛人数解码并校验配置。

只有 `GameSession` 创建事务成功后才冻结配置、参赛名单和座位，并关闭当前对局的进房准入。待开局记录必须与同一事务绑定，重复消费或过期令牌不得创建第二个会话。失败时保留房间和草稿，向房主展示可操作的错误原因。

## 6. 错误与并发策略

- 字段错误：在对应控件旁显示范围、依赖或容量约束，不允许提交无效配置。
- 权限错误：成员保持只读，服务端拒绝绕过客户端的写请求。
- 修订冲突：返回最新规范配置和新修订号，提示“规则已在另一设备更新”，房主可以重新应用本地调整。
- 房间状态冲突：对局开始、房间关闭或房间终态后，编辑和开局请求都被拒绝。
- 待开局失效：规则、成员、房主权属或房间状态变化会取消倒计时；客户端回到桌面并刷新最新规则，不继续本地倒计时。
- 网络失败：保留本地未提交编辑，显示重试；请求 ID 保证重试不会重复写入。
- 方案版本不兼容：方案标记不可用，提供“按当前游戏默认规则新建方案”。
- 服务端模块拒绝：展示游戏模块返回的稳定错误码和用户可理解的说明，不把 protobuf 或内部错误直接暴露给用户。

## 7. 测试与验收

### 7.1 游戏模块与协议

- 三款游戏默认值、边界值和依赖关系测试。
- 配置 envelope 编码/解码往返和规范化一致性测试。
- 实际参赛人数变化时的首次叫数、人数上限、容量和规则上限校验。
- 关闭规则后对应非法动作被服务端拒绝。
- malformed payload、配置载荷 hash、乘法溢出、未知协议版本和不可迁移方案测试。

### 7.2 房间与服务端

- 仅房主可以更新草稿和个人方案。
- 所有成员能读取同一份规则摘要和完整只读配置。
- 修订号冲突不会覆盖较新的配置。
- 切换游戏后各游戏草稿互不覆盖。
- StartGame 传入的配置被会话冻结，并在游戏桌和复盘中保持一致。
- 开局失败不会污染房间草稿；重复请求保持幂等。
- 房间版本、成员版本和房主转移/ownership fencing 的并发请求不会冻结错误的游戏或成员名单。
- 个人方案保存、覆盖、另存、删除、版本冲突和不可迁移状态均可验证。
- 房主转移后旧房主不能继续切换游戏、编辑草稿或写入个人方案，新房主可以在最新版本上继续操作。

### 7.3 Web 与视觉

- 规则抽屉在移动竖屏中支持拖拽展开、收起和独立滚动。
- 横屏改为右侧面板，玩家桌面和操作区不重叠。
- 非房主只读状态没有误导性的伪交互。
- 规则更新、冲突、倒计时取消和开局失败均有明确状态。
- 倒计时在重连、房主掉线、旋转屏幕和浏览器后台恢复后仍以服务端截止时间为准。
- Playwright 覆盖创建房间、切换游戏、编辑规则、跨成员同步、保存方案、应用方案和开始游戏。
- 对三款游戏分别做桌面宽屏、窄屏竖屏和横屏截图检查，验证安全区、焦点/键盘、文字不溢出、操作区不遮挡游戏桌、旋转后状态可恢复。
- 游戏桌和复盘断言使用的正是会话冻结快照，而不是当前房间草稿或用户默认方案。

## 8. 后续实现拆分建议

实现计划应按以下顺序拆分：

1. 定义共享规则 envelope、房间草稿修订和个人方案的服务端契约。
2. 为三款游戏补齐规则适配器的元数据、编码和客户端表单模型。
3. 接通 StartGame 的实际配置载荷和服务端冻结校验。
4. 实现房间页底部抽屉、横屏面板、切换游戏和只读投影。
5. 实现个人方案 CRUD、冲突恢复和默认规则恢复。
6. 补齐服务端集成测试、Web 单元测试和 Playwright 验收。
